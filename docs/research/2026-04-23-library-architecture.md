# Library Architecture — Research & Rebuild Proposal

> Draft: 2026-04-23
> Input for Sprint D (`master-improvement-plan.md`)

## Problem

Alfonso's critique: *"I think it's not thought out well at all. If anything, it should probably be pulled back quite a bit to just be files and photos or something like that until we figure out properly how to structure everything that's loaded in through MCPs and other sources of data properly."*

Today the library conflates:
- File uploads (PDF, DOCX, TXT, MD, XLSX)
- Photos (images from upload)
- Notes (rich text created in TipTap)
- MCP-sourced data (pulled via OAuth per provider)
- Auto-generated artifacts (code, docs from chat)
- Auto-summaries of conversations

All 6 types live in one `context_items` table with a `source_type` discriminator. The UI exposes them all in the same grid — users can't tell what's "mine" vs "from Drive" vs "an AI artifact".

## Research: how do peers structure this?

### Notion
- Root: Workspace → Pages (hierarchical, each page can contain databases).
- Each "database" has a schema; views (table, board, gallery, calendar).
- Imports from Drive/Slack/etc land as pages with a clear source badge.

### Obsidian
- Flat vault of Markdown files.
- Plugins add structure (Dataview for query, Canvas for visual).
- Source-of-truth = file system.

### Mem / Reflect
- Single stream of notes.
- Filters and backlinks form structure.
- "Sources" (web clips, imports) annotated in note metadata.

### Granola (our peer)
- Recordings → transcript → AI notes.
- Library is **by recording date**, not content type.

### Linear
- Database-per-team (issues, projects, cycles).
- No mixing types.

## Principle

**One UI per source-of-truth.** Mixing file uploads with MCP-pulled data in the same grid was a mistake — a user "deleting" an MCP item shouldn't delete remote state, but a file upload deletion is permanent.

## Proposed Structure (pulled back)

```
/library
  ├── Files           ← uploaded docs (PDF, DOCX, TXT, MD)
  ├── Photos          ← uploaded images + camera captures
  ├── Notes           ← TipTap documents created in Granger
  ├── Artifacts       ← code + docs generated by chat
  ├── Conversations   ← chat summaries (auto-embedded)
  └── Sources          ← MCP-pulled data, organized per-server
       ├── Google Drive
       ├── Linear
       ├── Granola
       ├── Slack
       └── …
```

**Key changes:**
1. **Top-level = source-of-truth type**, not mixed grid
2. **Sources sub-tree = read-only mirror** of MCP state; "delete" here means "hide from my view" not "delete remote"
3. **Files + Photos + Notes = user-owned** content (primary library)
4. **Artifacts + Conversations = AI-generated** (separate tab or filter)
5. **Each type gets tailored UI**: photos use grid+lightbox, files use list+preview, notes use TipTap, sources use provider-specific metadata

## Data Model Changes

Option A — keep `context_items`, add `library_section ENUM('file','photo','note','artifact','conversation','source')`:
```sql
ALTER TABLE context_items ADD COLUMN library_section TEXT NOT NULL DEFAULT 'source'
  CHECK (library_section IN ('file','photo','note','artifact','conversation','source'));
```

Option B — split into per-section tables (too invasive for now).

**Recommendation**: Option A. Computed from `source_type` via migration:
- `source_type='upload'` + mime image → `library_section='photo'`
- `source_type='upload'` + mime doc → `library_section='file'`
- `source_type='granger-note'` → `library_section='note'`
- `source_type='artifact'` → `library_section='artifact'`
- `source_type='conversation'` → `library_section='conversation'`
- everything else (google-drive, linear, slack, …) → `library_section='source'`

## UI Plan

1. Replace current `/context` top filter bar with section tabs: Files | Photos | Notes | Artifacts | Conversations | Sources.
2. Each tab has its own components:
   - Files: list with file-type icons + size + preview on hover
   - Photos: masonry grid + lightbox
   - Notes: grid of TipTap doc cards with title + excerpt
   - Artifacts: code cards with language badge + "Run" button
   - Conversations: chat cards with participant avatars + message count
   - Sources: provider-grouped with remote-only indicator and sync status
3. Global search stays cross-section.
4. Each tab has its own empty state + primary action ("Upload file", "Take photo", "New note", etc.).

## AI-Controllable Targets

New tools under this rebuild:
- `library_navigate(section)` — route to tab
- `library_upload(section, file)` — upload programmatically
- `library_create_note()` — create empty TipTap note
- `library_search(query, section?)` — scoped search
- `library_rename(id, title)`
- `library_tag(id, tags[])`

## Migration Path

1. **Phase 1** (this sprint): add `library_section` column + backfill via SQL.
2. **Phase 2**: update `/context` UI with tabs.
3. **Phase 3**: add missing AI tools.
4. **Phase 4**: deprecate old mixed-grid layout.

Expect tests added to `expect-plans/library.md` for each phase.
