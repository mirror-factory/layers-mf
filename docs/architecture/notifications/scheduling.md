# Scheduling System

> How scheduled actions work end-to-end in Layers: creation, execution, notification delivery, and management.

---

## Overview

Schedules let users set up recurring or one-shot AI tasks that run in the background. A schedule stores a prompt, runs it against the knowledge base on a cron interval, saves the result as a chat conversation, and notifies the user across all enabled channels.

Key properties:
- Users create schedules at `/schedules` via a date/time picker UI.
- The cron expression is generated internally -- users never see or write cron syntax.
- Execution happens server-side via `GET /api/cron/execute-schedules`.
- Results are saved as full chat conversations, viewable at `/chat?id={conversationId}`.
- Notifications fire across in-app bell, desktop browser, push (iOS/Android), and email.

---

## Schedule Creation

### User-facing UI

The schedule creation form at `/schedules` offers:

- **Name**: Display name for the schedule (required).
- **Prompt**: The AI instruction that runs on each execution (required). Stored in `payload.prompt` and `description`.
- **Frequency**: Date/time picker with presets:
  - **Once**: Runs a single time at the specified date/time. Stored as `once:{ISO_DATE}`.
  - **Daily**: Runs every day at a specific time. Generates a cron like `30 9 * * *`.
  - **Weekly**: Runs on specific days at a specific time. Generates a cron like `0 10 * * 1` (Monday 10 AM).
  - **Weekdays**: Runs Monday through Friday. Generates `0 9 * * 1-5`.
  - **Custom interval**: Hourly, every N hours, or specific cron. Generates the appropriate 5-field cron expression.
- **Max runs** (optional): Stops the schedule after N executions. Sets `max_runs` on the record.
- **Model** (optional): Override the default model. Stored in `payload.model`. Default is `google/gemini-3-flash`.

### Internal cron generation

The `schedule` field stores either:
- A 5-field cron expression (e.g., `0 9 * * 1-5` for weekdays at 9 AM).
- A one-shot prefix: `once:2026-04-08T14:00:00.000Z`.

The `calculateNextCron()` function in `src/lib/cron.ts` uses the `croner` library to compute the next run time from the cron expression. For one-shot schedules, the ISO date is used directly.

Human-readable descriptions are generated by `cronToHuman()` in the same module (e.g., `"Daily at 9:00 AM"`, `"Weekdays at 10:30 AM"`).

## Execution Flow

### Cron trigger

The executor lives at `src/app/api/cron/execute-schedules/route.ts` and is a `GET` handler.

**Authentication**: Requires `Authorization: Bearer {CRON_SECRET}` header. In demo mode (`DEMO_MODE=true`), auth is skipped.

**Note**: As of now, `execute-schedules` is configured with `maxDuration: 60` in `vercel.json` but is NOT listed in the Vercel `crons` array. It needs to be triggered externally (e.g., via an external cron service, Vercel cron config update, or manual invocation). The comment in the code says "runs every minute" but that requires adding it to the crons array.

### Execution steps (per schedule)

For each due schedule (up to 10 per run):

1. **Find due schedules**: Query `scheduled_actions` where `status = 'active'` and `next_run_at <= now()`.

2. **Create conversation**: Insert into `conversations` with `initiated_by: "schedule"` and `schedule_id`. This links the conversation back to the schedule.

3. **Save user message**: Insert the prompt as a `user` role message in `chat_messages` with `channel: "schedule"`.

4. **Run AI**: Call `generateText()` with:
   - Model: `google/gemini-3-flash` via AI Gateway (fast and cheap for background tasks).
   - System prompt: Granger identity with instructions to use search_context and be concise.
   - Tools: `search_context` only (searches knowledge base via hybrid vector + BM25). No sandbox, no integrations, no artifact creation -- these are too expensive/slow for background runs.
   - Stop condition: `stepCountIs(5)` -- limits to 5 tool-calling steps maximum.

5. **Save AI response**: Insert the response text as an `assistant` role message in `chat_messages`.

6. **Update schedule metadata**:
   - `last_run_at`: Current timestamp.
   - `run_count`: Incremented by 1.
   - `next_run_at`: Calculated from cron expression (null for completed one-shots).
   - `status`: Set to `"completed"` if one-shot or `max_runs` reached; otherwise stays `"active"`.
   - `error_message`: Cleared on success.
   - `last_conversation_id`: The new conversation ID.

7. **Notify user**: Calls `notify()` with type `schedule_complete` (see Notification Delivery below).

### On failure

If any step throws:
- The error message is written to `error_message` on the schedule row (does not change status).
- `notify()` is called with type `system_alert`, title `"Schedule failed: {name}"`, link to `/schedules`.
- The executor continues processing remaining schedules (errors do not halt the batch).

### On-demand execution

Users can trigger a schedule immediately via the "Run Now" button, which calls `POST /api/schedules/execute` with `{ scheduleId }`. This follows the same execution and notification flow but:
- Uses the authenticated user's context (not CRON_SECRET).
- Has `maxDuration: 120` (longer timeout for interactive use).
- Currently inserts directly into `notifications` table instead of using `notify()`, so push and email are NOT sent for on-demand runs. This is a known gap.

## Available Tools During Scheduled Runs

Only one tool is available:

| Tool | Description |
|------|-------------|
| `search_context` | Searches the org's knowledge base (context_items + chunks) via hybrid vector + BM25 search. Returns title, snippet, source_type, content_type. |

Tools intentionally excluded from scheduled runs:
- `sandbox` / `run_code` -- too slow and expensive for background
- Integration tools -- require user OAuth context
- `propose_action` -- approvals need interactive context
- `create_artifact` / `edit_artifact` -- no interactive preview

## Notification Delivery

After successful execution, `notify()` fires across all channels:

| Channel | Behavior |
|---------|----------|
| In-app bell | Always. Inserts row in `notifications` table. Bell polls every 30s. |
| Desktop browser | Fires when the notification bell detects the new row (requires tab open + permission granted). |
| Push (iOS) | Sends via APNs to all registered device tokens. Needs APNs p8 key configured. |
| Push (Android) | Built but skipped -- needs Firebase credentials. |
| Email | Sends via Resend if `digest_enabled` preference is true. From: `notifications@mirrorfactory.ai`. |
| Sound | Ping plays in browser if sound is enabled in /settings/notifications. |

On failure (type `system_alert`): in-app and push fire, but email does not (system_alert is not mapped to any email preference).

## Schedule Management

### API Routes

| Route | Method | Description | Auth |
|-------|--------|-------------|------|
| `/api/schedules` | GET | List all schedules for the user's org (max 100, newest first) | Supabase |
| `/api/schedules` | POST | Create a new schedule. Required: `name`, `schedule`. Optional: `description`, `prompt`, `action_type`, `target_service`, `payload`, `max_runs`, `model`. | Supabase |
| `/api/schedules/[id]` | PATCH | Update a schedule. Fields: `status` (active/paused), `name`, `prompt`, `schedule`. Recalculates `next_run_at` when schedule changes. | Supabase |
| `/api/schedules/[id]` | DELETE | Delete a schedule permanently. | Supabase |
| `/api/schedules/execute` | POST | Execute a schedule on-demand ("Run Now"). Body: `{ scheduleId }`. | Supabase |
| `/api/cron/execute-schedules` | GET | Cron executor. Finds and runs all due schedules. | Bearer token (CRON_SECRET) |

### PATCH operations

- **Pause**: `PATCH /api/schedules/{id}` with `{ "status": "paused" }`. Schedule stops executing until resumed.
- **Resume**: `PATCH /api/schedules/{id}` with `{ "status": "active" }`. Resumes from next calculated run time.
- **Edit prompt**: `PATCH /api/schedules/{id}` with `{ "prompt": "new prompt" }`. Updates both `description` and `payload.prompt`.
- **Change schedule**: `PATCH /api/schedules/{id}` with `{ "schedule": "0 10 * * 1-5" }`. Recalculates `next_run_at`.
- **Rename**: `PATCH /api/schedules/{id}` with `{ "name": "New Name" }`.

### View last run

The `last_conversation_id` field on the schedule links to the most recent execution. Navigate to `/chat?id={last_conversation_id}` to see the full AI conversation including any tool calls and search results.

## Database: scheduled_actions Table

| Column | Type | Description |
|--------|------|-------------|
| `id` | uuid | Primary key |
| `org_id` | uuid | Organization the schedule belongs to |
| `created_by` | uuid | User who created the schedule |
| `name` | text | Display name |
| `description` | text | The prompt text (also stored in payload.prompt) |
| `action_type` | text | Type of action (default: `"custom"`) |
| `target_service` | text | Optional target service identifier |
| `payload` | jsonb | Arbitrary data. Key fields: `prompt` (the AI instruction), `model` (optional override) |
| `schedule` | text | Cron expression or `once:{ISO_DATE}` |
| `next_run_at` | timestamptz | When the schedule should next execute. Null when completed. |
| `last_run_at` | timestamptz | When the schedule last executed |
| `run_count` | integer | Number of times the schedule has executed (default: 0) |
| `max_runs` | integer | Optional maximum number of executions. Schedule completes when reached. |
| `status` | text | One of: `active`, `paused`, `completed` |
| `error_message` | text | Last error message (cleared on successful run) |
| `last_conversation_id` | uuid | ID of the most recent conversation created by this schedule |
| `created_at` | timestamptz | Creation timestamp |

---

## Related Tables

| Table | Relationship |
|-------|-------------|
| `conversations` | Created per execution. `initiated_by = 'schedule'`, `schedule_id` links back. |
| `chat_messages` | User + assistant messages for each execution. `channel = 'schedule'`. |
| `notifications` | Created after each execution via `notify()`. |
| `notification_preferences` | Checked for email delivery decisions. |
| `device_tokens` | Checked for push delivery. |

## Testing

### Local testing

Trigger the executor manually:

```bash
curl -s http://localhost:3204/api/cron/execute-schedules \
  -H "Authorization: Bearer test"
```

This works when `DEMO_MODE=true` (bypasses auth) or when the Bearer token matches `CRON_SECRET`.

### Creating a test schedule

```bash
# Create a schedule that runs immediately (once, in the past)
curl -s http://localhost:3204/api/schedules \
  -H "Content-Type: application/json" \
  -H "Cookie: <auth_cookie>" \
  -d '{
    "name": "Test Schedule",
    "prompt": "Summarize the latest meeting notes",
    "schedule": "once:2026-04-01T00:00:00.000Z"
  }'

# Then trigger the executor
curl -s http://localhost:3204/api/cron/execute-schedules \
  -H "Authorization: Bearer test"
```

### Verifying results

1. Check the response from the executor -- it returns `{ executed: N, results: [...] }` with conversation IDs.
2. Check the `notifications` table for the `schedule_complete` notification.
3. Navigate to `/chat?id={conversationId}` to see the AI's output.
4. Check the `scheduled_actions` table for updated `last_run_at`, `run_count`, and `last_conversation_id`.
